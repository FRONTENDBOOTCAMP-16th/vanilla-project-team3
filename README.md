# 브랜치 컨벤션

| 브랜치 명                    | 목적      | 설명                                                                                                       |
| :--------------------------- | :-------- | :--------------------------------------------------------------------------------------------------------- |
| `main`                       | 배포용    | 항상 안정적인 프로덕션(운영) 상태 유지. 직접 코딩 및 푸시(Push) 절대 금지.                                 |
| `develop`                    | 개발 통합 | 다음 릴리즈를 위해 각 기능 브랜치를 통합하는 곳. **반드시 PR 및 최소 1명 이상의 코드 리뷰 승인 후 Merge.** |
| `feature/{이슈번호}-{설명}`  | 기능 개발 | 새로운 기능 개발 시 `develop`에서 분기하여 사용. (예: `feature/#12-login-api`)                             |
| `fix/{이슈번호}-{설명}`      | 버그 수정 | 개발 중 발견된 일반적인 버그 수정용 브랜치.                                                                |
| `hotfix/{이슈번호}-{설명}`   | 긴급 수정 | **운영(main) 배포 후** 발생한 치명적인 장애 처리 시 사용. `main`에서 분기.                                 |
| `refactor/{이슈번호}-{설명}` | 리팩토링  | 비즈니스 로직 변화 없이 코드 구조나 성능을 개선할 때 사용.                                                 |
| `chore/{이슈번호}-{설명}`    | 설정/환경 | 패키지 설치, 웹팩/바이트(Vite) 빌드 설정 변경 등 코드 외적 작업 시 사용.                                   |

# 주요 기능

## 1. main 페이지

- 날씨 / 기분 아이콘 선택 후 비회원확인버튼 클릭으로 페이지 이동
- 로그인 버튼 클릭시 로그인 페이지로 이동
- 회원가입 버튼 클릭시 회원가입 페이지로 이동

## 2. 문장 페이지

- 책 제목, 작가, 문장(한글자씩 생성) 화면에 표시
- 찜 버튼으로 해당 문장 my 페이지에 저장(최대6개)-우선순위2
- 공유 버튼으로 외부로 공유하기(URL방식)-우선순위2

## 3. 로그인 페이지

- 아이디 / 비밀번호 입력
- 회원가입 버튼으로 회원가입 페이지로 이동
- 홈버튼으로 메인 db 페이지로 이동

## 4. 회원가입 페이지

- 아이디(중복확인) / 비밀번호 / 이메일 입력
- 홈버튼으로 메인 db 페이지로 이동

## 5. my 페이지

- 비밀번호 변경
- 찜 리스트 확인
- 홈버튼으로 메인 db 페이지로 이동

# 트러블 슈팅

- 코드 병합
  - 문제 상황: 공유 기능 수정 및 팀원 코드 병합 후, 결과 페이지 진입 시 데이터 로드에 실패함. "데이터가 존재하지 않습니다"라는 알림과 함께 메인 페이지(index.html)로 강제 리다이렉트되는 현상 발생.일부 로직에서 유저 데이터를 불러오기 전 호출이 일어나 null 혹은 undefined가 전달되어 추천 로직이 정상 작동하지 않음.

  - 원인 파악 : 모든 페이지에서 공통으로 사용하는 main.js가 \_popup.js를 불러오고, \_popup.js가 다시 특정 페이지 전용 로직인 result.js를 참조함.
    관심사 분리 실패 : result.js 내부에 UI 렌더링 로직과 데이터 처리(비즈니스 로직)가 뒤섞여 있어, 특정 함수 호출 시 의도치 않게 페이지 전체의 초기화 로직이 간섭을 받음.
  - 해결 방법 : 비즈니스 로직 분리: result.js에 묶여 있던 데이터 계산 및 API 통신 로직을 별도의 bookService.js로 분리함.
    \_popup.js가 더 이상 result.js를 직접 참조하지 않도록 수정하고, 필요한 로직은 분리된 bookService.js에서 가져오도록 변경하여 의존성 오염을 제거함.
    bookService.js: 순수 데이터 처리 및 API 통신 담당 (페이지 종속성 없음).
    result.js: result.html 전용 UI 제어 및 렌더링 담당.
  - 배운 점 : 이론으로 접한 원칙을 코드에 적용하려고 노력했으나 기능 간의 결합도가 높아지는 스파게티 코드를 경험하게 되고 이를 해결하는 과정에서 초기 설계 단계에서 명확한 역할 분리가 얼마나 중요한지 깨닫는 계기가 되었다.꼬여있는 로직을 피드백 받은대로 분리하고 재구성하는 리팩토링 과정을 통해서 유지보수가 용이한 설계를 어떻게 하는지 알 수 있는 경험을 할 수 있었다.또한 동일한 기능을 중복 구현하거나 코드 충돌이 발생하는 문제를 겪으면서 효율적인 협업 및 분업을 위해서는 정말 세밀한 의논과 협의 과정이 필수라는것을 깨닫게 되었다. 스파게티 코드를 경험하고 관심사의 분리를 진행하면서 앞으로는 구조를 어떻게 짜야할지 경험을 할 수 있어서 정말 좋은 기회였다. 팀원들과 협업하는 과정에서 각자 코드를 작업하더라도 코드가 중복되거나 같은 곳에서 작동해 충돌이 일어나게 되는 문제들을 겪으면서 협업은 정말 많은 의논과 협의를 거쳐서 진행해야한다는것도 알게되었다.

- SPA
  - 문제 상황: SPA에서 네비게이션을 눌렀을때 페이지 전환을 해야하는 상황.페이지 전환을 하기위해 이때까지 작업을 한 페이지들을 index.html에 전부 몰아넣어야했고, 버튼을 누르면 class하나로 페이지를 제어해야되었다.그렇게 되다보니 페이지 앞/뒤로 이동이 힘들게 되었고, 브라우저가 지원하는 페이지 이동 기능이 적용하지 않게 되었다.
  - 원인파악 : SPA이 원인이였다.
    SPA이여서 페이지 전환을 하려면 따로 홈페이지를 뜯어서 기능부터 만들어 제작했어야 했다.
  - 해결 방법 :
    해당 기능을 만드는 시간도 이해도 부족했다.
    처음에는 페이지 수가 적으니 SPA으로 진행하면 된다고 가볍게 생각하여 진행했는데, SPA의 장단점을 알고 우리 프로젝트에 적합하지 않다는 점을 알게되었다. 그래서 당장 SPA을 고집하기보다 MAP으로 변경하여 진행하는것이 낫다고 결론이 나어 기틀을 변경하고 다시 작업을 진행하기로 하였다.
  - 배운 점 : 페이지의 구조의 장단점을 확실히 알고 이해하여, 페이지에 적용해야한다는 점을 배웠다.단순하게 페이지 수가 적으니 SPA을 쓰면된다고 생각하는게 아니라, SPA과 MAP의 장점과 단점이 있으니 잘 파악하고 써야한다는것을 알게되었다.
    그리고 이번 계기로 SPA과 MAP이 어떤 차이가 있고, 어떤 장단점이 있는지 알게되는 계기가 되었다.

- 찜하기 더미데이터 추가
  - 문제 상황 : 책 찜하기 클릭 시 마이페이지 책 리스트에 더미데이터가 추가됨
  - 원인 파악 : 회원가입 시 API 전송 데이터에 heart, viewed 필드가 없어 더미데이터가 생성됨
  - 해결 방법 : heart: [], viewed: [] 빈 배열을 초기값으로 API에 함께 전송
  - 배운 점 : 데이터 저장 시 기본값을 명시하지 않으면 예상치 못한 값이 들어갈 수 있음을 깨달았고, 데이터 구조의 초기 설계를 탄탄히 해야 한다는 것을 배웠다.
- 동일 아이디 동시 로그인
  - 문제 상황 : 서로 다른 창에서 동일 아이디로 동시 로그인이 가능함
  - 원인 파악 :
  - 중복 로그인을 차단하는 로직이 구현되어 있지 않았음
  - 페이지 로드 시 localStorage를 초기화하는 코드로 인해 다른 탭의 세션이 삭제되는 부작용이 있었음
  - 해결 방법 :
  - API에 isLoggedIn 필드를 추가하여 동시 로그인 차단
  - isLoggedIn: true 상태일 때 무조건 차단하던 방식을 confirm으로 변경하여 사용자가 강제 로그인 여부를 선택할 수 있도록 개선
  - initSession()의 else 블록에서 localStorage.removeItem() 제거로 다른 탭 세션 삭제 부작용 방지
  - 한계 : MockAPI 무료버전 제약으로 브라우저 종료 시 isLoggedIn을 false로 업데이트할 수 없어, 재접속 시 강제 로그인 confirm으로 대응함
  - 배운 점 : 동일 아이디 동시 로그인은 개인정보 보호 측면에서도 반드시 차단해야 한다는 것을 피드백을 통해 깨달았다. 또한 세션 관리 시 localStorage 초기화가 다른 탭에 영향을 줄 수 있다는 것을 배웠다.

- 회원가입 중복 검사 404 에러
  - 문제 상황 : 회원가입 중복 검사 시 콘솔에 404 에러가 발생함
  - 원인 파악 :
  - 브라우저는 404 응답 자체를 콘솔에 항상 표시하며, JavaScript로는 막을 수 없음
  - 단, 코드 동작 자체는 정상 (중복 없으면 가입 정상 진행)
  - MockAPI가 404를 반환할 때 checkResponse.json()을 실행하면 응답 body가 JSON이 아닌 경우 파싱 에러가 추가로 발생할 수 있었음
  - 해결 방법 : checkResponse.ok 조건을 추가하여 404 응답 시 json() 파싱을 시도하지 않고 바로 빈 배열 []을 반환하도록 수정
  - 배운 점 : 콘솔에 표시되는 에러에 너무 집착하지 않아야 한다는 것을 배웠다. 코드가 정상 동작하더라도 브라우저 자체의 동작으로 인해 에러가 표시될 수 있으며, 이는 개발자가 제어할 수 없는 영역임을 깨달았다.

- 문제 4. 경고 메시지로 인한 input 레이아웃 무너짐
  - 문제 상황 : 로그인, 회원가입 페이지에서 경고 메시지 표시 시 input 박스 레이아웃이 밀리는 현상 발생
  - 원인 파악 : hidden 속성을 사용하면 요소가 DOM에서 공간 자체를 차지하지 않아 숨겨질 때 레이아웃이 밀림
  - 해결 방법 :
  - hidden 대신 visibility: hidden/visible로 변경하여 공간은 유지하면서 텍스트만 보였다 안 보였다 하도록 수정
  - CSS: .warning-block의 기본값을 visibility: hidden으로 설정
  - HTML: hidden 속성 제거
  - JS: .hidden = true/false 를 .style.visibility = 'hidden'/'visible' 로 전부 교체
  - 배운 점 : hidden과 visibility의 차이를 이해하게 되었다. 요소를 숨길 때 공간까지 없애야 하는지, 공간은 유지하면서 안 보이게만 해야 하는지를 구분하여 상황에 맞는 속성을 사용해야 한다는 것을 배웠다.
